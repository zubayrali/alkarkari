# i18n Design — Karkari Wiki

**Date:** 2026-07-02
**Status:** Draft v2 (isolated-builds architecture; v1's fumadocs-i18n routing rejected)

## Goal

Offer the site in multiple locales with **one Obsidian vault per language** and
**complete isolation between locales**: no shared routing, no cross-locale
fallback, no overlapping search/graph/tags. Each locale is its own static
build of the same codebase; switching language means pointing at a different
build. Simple and foolproof by construction — a locale cannot leak into
another because they never share a build.

## Architecture: N isolated builds, one deploy

```
                    ┌ SITE_LANGUAGE=en, vault EN ─→ next build ─→ out-en ┐
same codebase  ──── ├ SITE_LANGUAGE=cn, vault CN ─→ next build ─→ out-cn ├─→ stitched
                    └ SITE_LANGUAGE=ar, vault AR ─→ next build ─→ out-ar ┘   GitHub Pages site

deployed site:
  /                 → tiny static locale chooser (hand-written, not Next)
  /en/**            → the entire English site (its own search, graph, tags, bases)
  /cn/**            → the entire Chinese site
  /ar/**            → …
```

- **No fumadocs i18n, no `app/[lang]`, no middleware, no `defineI18n`.** The
  app stays exactly the single-locale app it is today. `SITE_LANGUAGE` (env,
  build-wide) stays — it stops being a compromise and becomes the mechanism.
- Locale prefix comes from `PAGES_BASE_PATH=/<repo>/<locale>` — the existing
  basePath machinery, nothing new. Every asset, search index, RSS, sitemap,
  OG image is automatically scoped under `/<locale>/`.
- Isolation is total: each build sees one vault's content and nothing else.
  Search cannot return another language's pages. The graph cannot contain
  them. There is no "complex routing scenario" because there is no routing.

**Rejected (v1):** fumadocs i18n routing (`app/[lang]`, shared loader,
`fallbackLanguage`). More capable (same-URL language switching, English
fallback for untranslated pages, hreflang pairing) but every one of those
capabilities is cross-locale coupling — exactly what this design forbids. The
route-tree migration also risked the fragile view-transition/sidebar CSS
(ADR-0007/0009) for no isolation benefit.

## Locale set

| Phase | Locales | Notes |
|---|---|---|
| Launch | `en`, `cn` | UI strings already exist for both |
| Next | `fr`, `es`, `tr`, `ur` | LTR — strings + a vault |
| Gated | `ar` | Blocked on the RTL work package (§7) |

Adding a locale costs: one `lib/locale.ts` entry, one line in the locales
list (`lib/locales-manifest.ts`, also consumed by CI), one vault. Nothing else.

## Repo layout & generation

Committed per-locale trees; the live `content/` + `public/` become staging
directories (gitignored) populated at build time:

```
locales/
  en/content/**   en/public/**      ← pnpm generate --locale=en writes here
  cn/content/**   cn/public/**
content/  public/                   ← staged copy of ONE locale (gitignored)
```

- `.env`: `OBSIDIAN_VAULT_PATH_<LOCALE>`, `GENERATE_INCLUDE_<LOCALE>`
  (existing unsuffixed vars = default locale, kept during migration).
- `pnpm generate --locale=cn` reads the CN vault, cleans and rebuilds only
  `locales/cn/`. Teams commit only their `locales/<locale>/` subtree —
  no cross-team conflicts by construction.
- `pnpm stage <locale>` copies `locales/<x>/content` → `content/` and
  `locales/<x>/public` → `public/` (dev + CI prebuild). `pnpm dev` for a
  locale = stage, then dev as today.
- Wikilinks, aliases, tags, backlinks, note embeds, bases notes-index: all
  resolve within the staged tree — i.e. within the locale. Unchanged code.

**No slug parity requirement.** Vaults may organize content entirely
differently (an Arabic team can use Arabic filenames). The switcher (§5)
doesn't depend on matching URLs. Parity remains a *recommendation* for teams
that want deep-link swapping later, but nothing enforces or assumes it.

## Root chooser & language switcher

- **Root `/`**: a small hand-written static page (checked into
  `deploy/root/index.html` + assets, copied verbatim during stitch): the
  muraqqaʿa star, one link per locale (own language name), remembers choice in
  `localStorage` and auto-forwards on revisit; `navigator.languages` picks the
  first-visit default. Plain `<a>` links as fallback — works with JS disabled.
- **Switcher in the site chrome**: a simple nav link set (from
  `lib/locales-manifest.ts`) pointing to each locale's **home** (`/<locale>/`).
  It is a dumb absolute link to another build — no slug mapping, no lookup.
  If two vaults happen to share a slug, readers can edit the URL; we don't
  promise it.
- **Old URLs** (`/<repo>/dictionary/wird` from the pre-i18n site): the stitch
  step emits meta-refresh stubs redirecting old top-level paths to
  `/en/<same-path>`. Removable after a deprecation window.
- Custom 404 (GitHub Pages `404.html`, per stitch root): offers links to each
  locale home.

## Deployment (single repo, matrix build)

`deploy.yml` becomes: matrix job over the locales manifest — each leg runs
`pnpm stage <locale>` → quality gates → `pnpm build` with
`PAGES_BASE_PATH=/<repo>/<locale>`, `SITE_LANGUAGE=<locale>` → upload
`out-<locale>` artifact. A stitch job assembles
`site/ = deploy/root/* + en/ + cn/ + …` and deploys once. One Pages site,
one custom-domain story, N isolated builds inside it.

A broken locale build fails the whole deploy (shared gate). Acceptable at this
team size; if it ever hurts, per-locale Pages projects are a drop-in escape
hatch — that's the virtue of full isolation.

## UI strings

`lib/locale.ts` keyed by `SITE_LANGUAGE` — the mechanism already in place; the
2026-07 housekeeping (all homepage copy in the `home` block) already closed
the string gaps. Each new locale adds its entry. `htmlLang` feeds
`<html lang>`; the manifest adds `dir` (`ltr`/`rtl`) feeding `<html dir>`.

## Per-locale details

| Concern | Resolution |
|---|---|
| Search | per-build index; `searchLanguage` per locale (CJK tokenization for `cn` — accept naive forward tokenize at launch, revisit) |
| Cusdis | one app-per-locale **or** locale-prefixed `pageId` (`en:dictionary/wird`) — prefix chosen; threads never collide |
| Open in Obsidian | vault name per locale (`lib/obsidian.ts` reads a manifest field) |
| OG / RSS / sitemap | emitted per build under `/<locale>/` automatically; root gets a `robots.txt` + sitemap-index listing each locale sitemap |
| SEO pairing | no hreflang between locales at launch (needs cross-locale knowledge we've excluded); each locale indexes independently. Revisit only if parity is adopted |
| Review widget / reader prefs (localStorage) | **no prefixing needed** (decided at implementation): review entries are keyed by content hash and graph-visited by URL (basePath included), so locales never collide; sidebar/theme prefs sharing across locales is benign, arguably desirable |

## RTL work package (gates `ar`)

Unchanged from v1: `<html dir="rtl">` from the manifest; logical-properties
audit of custom CSS (sidenotes margin engine, properties panel, canvas
controls, reader-mode exit bar, karkari theme, nav progress); Amiri as body
face for `ar`; explicit RTL testing of sidenotes placement and graph labels.

## Out of scope

- Same-slug deep-link language switching, hreflang pairing, cross-locale
  fallback or transclusion — all reintroduce coupling; revisit only with a
  concrete need and the parity convention adopted.
- Translation-management tooling / machine translation.

## Open questions

   the chooser?
2. Does `en` also live under `/en/` (proposed — uniform) or stay at the root
   as today (saves redirects, breaks uniformity)?

   free tier for ≤4 locales; revisit caching if slow.
