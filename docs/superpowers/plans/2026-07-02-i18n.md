# i18n Implementation Plan — Isolated Builds

**Date:** 2026-07-02
**Spec:** `docs/superpowers/specs/2026-07-02-i18n-design.md` (v2)
**Status:** Ready after spec review

The app code barely changes — this is mostly generation, repo layout, and CI
work. Phased so the site is deployable after every phase; each phase ends with
`pnpm types:check && pnpm lint && pnpm test && pnpm build` plus listed checks.

## Phase 1 — Repo layout + staging (single locale, no visible change)

1. `lib/locales-manifest.ts` — `[{ locale: 'en', label: 'English', dir: 'ltr', vaultEnv: 'OBSIDIAN_VAULT_PATH_EN' }]`; consumed by app + scripts + CI.
2. Move `content/` → `locales/en/content/`, generated `public/` assets →
   `locales/en/public/`; gitignore live `content/` + `public/`.
3. `pnpm stage <locale>` script (copy staging); wire into `predev`/`prebuild`
   convenience (`pnpm dev:en`).
4. `scripts/generate.ts` — `--locale` flag; suffixed env vars
   (`OBSIDIAN_VAULT_PATH_EN`, fallback to unsuffixed); write/clean under
   `locales/<locale>/` only.
5. CI: stage `en` before gates + build (still a single build, still deploys to
   the repo root as today).

**Verify:** site byte-identical to before (diff `out/`); `pnpm generate
--locale=en` touches only `locales/en/`; dev flow documented in CLAUDE.md.

## Phase 2 — Matrix deploy under locale prefixes

1. `deploy.yml`: matrix over the manifest; per-leg `pnpm stage` → gates →
   `PAGES_BASE_PATH=/<repo>/<locale>` `SITE_LANGUAGE=<locale>` `pnpm build` →
   artifact.
2. Stitch job: `deploy/root/` (hand-written chooser `index.html`, `404.html`,
   `robots.txt`, sitemap index) + each `out-<locale>` under `/<locale>/`;
   single Pages deploy.
3. Legacy redirect stubs: old unprefixed paths → `/en/<path>` (generated in
   the stitch step from the en build's route list).
4. Locale switcher links in nav/footer from the manifest (dumb links to each
   locale home). Locale-prefix the Cusdis `pageId` and localStorage keys
   (review widget, sidebar, graph-visited, reader prefs).

**Verify:** `/` chooser works with and without JS; `/en/**` fully functional
under the deeper basePath (search, bases fetches, canvas JSON, OG, RSS);
old URLs redirect; 404 page offers locale homes.

## Phase 3 — Second locale (`cn`)

1. Add `cn` to the manifest; CN vault + `OBSIDIAN_VAULT_PATH_CN`;
   `pnpm generate --locale=cn`; commit `locales/cn/`.
2. Confirm `searchLanguage: 'chinese'` tokenization is acceptable (open
   question 3 in spec); adjust flexsearch options if not.
3. CI matrix picks `cn` up automatically from the manifest.

**Verify:** `/cn/**` is a complete isolated site — its search never returns
English pages, graph/tags/bases contain only CN content; `en` build unchanged
(artifact diff clean); switcher round-trips.

## Phase 4 — Scale-out + RTL, then `ar`

1. `fr`/`es`/`tr`/`ur` locale strings in `lib/locale.ts` as translations
   arrive — manifest line + strings + vault each.
2. RTL package: `dir` from manifest onto `<html>`; logical-properties audit
   (sidenotes engine, properties panel, canvas controls, reader-mode bar,
   karkari theme, nav progress); Amiri body face for `ar`.
3. Add `ar`.

**Verify:** adding a throwaway locale touches only manifest + strings + vault
and takes <1h; `ar` renders mirrored correctly; LTR locales unaffected.

## Risks

- **Deeper basePath** (`/<repo>/<locale>`) exercises every hardcoded-path
  assumption: bases/canvas/excalidraw fetch URLs, search-index fetch, OG
  routes. All go through `NEXT_PUBLIC_BASE_PATH` today — Phase 2 verification
  must hit each surface.
- **Stitch step correctness** (redirect stubs, sitemap index) is new bespoke
  CI code — keep it a small tested script (`scripts/stitch-deploy.ts`), not
  inline YAML.
- **Shared origin localStorage** across locale builds — prefixing is easy to
  miss in one consumer; grep for `localStorage` in Phase 2 and centralize key
  construction in one helper.
- **N× build time** per deploy — matrix parallelizes; content is prebuilt in
  the repo, so each leg is compile-only. Revisit only if it drags.
