# Page navigations crossfade via Next 16's native View Transitions, not a third-party library

Fumadocs has no built-in support for animated page transitions — there are no mentions of "view transition" in `fumadocs-ui` or `fumadocs-core`. Next.js 16.2.7 (the installed version) ships `experimental.viewTransition`, which enables React's `<ViewTransition>` component and wraps App Router navigations in `document.startViewTransition()`. We considered `next-view-transitions` (a community wrapper around the raw browser API), but it's superseded by the native integration on Next 16 and would add a dependency for something the framework now does itself. The native flag is also a single, fully reversible config toggle, consistent with ADR-0006's "bare, out of the box" direction for this app's layout structure.

`next.config.mjs` sets `experimental: { viewTransition: true }`. `app/(docs)/[...slug]/page.tsx` wraps the `<DocsTitle>`...`<DocsBody>` content in `<ViewTransition name="docs-content" share="auto" enter="auto" default="none">`, giving the content area a stable identity so old and new content crossfade on navigation; `default="none"` suppresses animation on first render. Without anchoring, the View Transitions API would also crossfade the persistent `DocsLayout` sidebar (sticky, restored in ADR-0006), causing it to flicker on every docs-to-docs navigation. `app/(docs)/[...slug]/layout.tsx` gives the sidebar a stable identity via `sidebar={{ style: { viewTransitionName: 'site-sidebar' } }}`, and `app/global.css` suppresses animation for `::view-transition-group/old/new(site-sidebar)` so it stays static. `app/global.css` also adds the standard `prefers-reduced-motion: reduce` override, zeroing animation duration/delay for all `::view-transition-*` pseudo-elements — required for a production-ready implementation, not optional. The home page (`/`, `HomeLayout`) initially shipped with no `<ViewTransition>` at all, on the assumption that the browser's default root crossfade would cover home↔docs navigation. In practice React only invokes `document.startViewTransition()` for a commit if at least one `<ViewTransition>` component is mounting, unmounting, or updating in that commit — with zero `<ViewTransition>` anywhere in the home tree, navigating to or from `/` never triggered a transition at all. A first fix gave `app/(home)/page.tsx` its own named boundary (`<ViewTransition name="home-content" enter="auto" exit="auto" default="none">`), which produced the correct `vt-*` SSR markup but still rendered as an instant snap in the browser — `home-content` and `docs-content` are unmatched names living in entirely different layout subtrees (`HomeLayout` vs `DocsLayout`), so each only gets an independent, unpaired enter/exit rather than a real crossfade.

The working fix instead gives `app/(home)/page.tsx`'s wrapper the **same name** as the docs page's: `<ViewTransition name="docs-content" share="auto" enter="auto" default="none">`. Because both sides of a home↔docs navigation now produce a `vt-name="docs-content"` element, React pairs them as a single "update" transition with `share="auto"` — the exact mechanism already proven to crossfade docs↔docs navigation — rather than two independent, unpaired enter/exit animations. This gives home↔docs navigation (in both directions, including back-navigation to `/`) the same crossfade as docs↔docs.

## Sidebar exit on docs→home (follow-up)

The `docs-content` pairing crossfades the content area, but the named `site-sidebar` remained a problem on docs→home specifically. The original rule suppressed its animation entirely (`::view-transition-group/old/new(site-sidebar) { animation: none }`) to keep it static during docs↔docs. That works while the sidebar is *matched* (it exists on both sides), but home has no sidebar — so on docs→home the old sidebar snapshot was unmatched and, with no animation, sat at full opacity for the whole transition and then popped out of existence (symptom: "content becomes home, but the sidebar hangs then pops").

First attempt kept `animation: none` on the *new* snapshot and the group, animating only the *old* one, on the theory that during docs↔docs the instant new would occlude the fading old. It didn't reliably take — the safer model is to not special-case matched vs unmatched at all. **Both** `::view-transition-old/new(site-sidebar)` now simply **fade** (`vp-sidebar-out`/`vp-sidebar-in`, 0.2s). On docs↔docs that's a crossfade between two near-identical sidebars (only the active-item highlight differs), which is imperceptible; on docs→home the old one fades cleanly out with no new to pair. The name is also pinned via `#nd-sidebar { view-transition-name: site-sidebar }` in `app/global.css` so it doesn't depend on fumadocs plumbing the layout `style` prop onto the `<aside>`. The `prefers-reduced-motion` block already zeroes it via the `(*)` selector.

Note: `::view-transition-*` rules are aggressively cached by the dev server — changes here need a clean restart (clear `.next`) and a hard refresh to take effect; HMR alone keeps serving the previous transition CSS.

## Base-table "magic move" + content fade-through

With base pages now rendering as full-width data tables, the single `docs-content` cross-dissolve looked muddy — a wide table and a narrow article both painted at 50% opacity mid-transition. Two layered changes address it:

**Fade-through with rise.** `::view-transition-old(docs-content)` fades out fast (0.16s) and `::view-transition-new(docs-content)` fades in *after* a short delay while rising 10px (`vp-content-in`, 0.26s, delay 0.1s). Old and new no longer overlap at half-opacity; the navigation reads as directional rather than a dissolve. Applies to every content navigation, not just base pages.

**Magic move (shared-element morph).** A base-table row title morphs into the destination entry's `<h1>`. `lib/transition-name.ts`'s `entryTransitionName(path)` produces a stable `entry-<sanitised-path>` `view-transition-name` from a path string; the base row uses `NoteRecord.slug` and the entry page's `DocsTitle` uses `page.url`. Because both ends emit the same name, the browser pairs them and runs the default group morph (position + size + crossfade) — the row title appears to fly into the heading. The name is unique per slug, so it only ever pairs a row with its own entry; on navigations where no row was clicked (article→article), each H1 carries a different `entry-*` name, is unmatched, and simply fades like the rest of the content. Unmatched sibling rows on the source table likewise fade out individually — acceptable since the whole table is fading anyway. Scoped to the **table** view (`bases-view-table.tsx`); gallery/list don't morph. Reduced-motion zeroes all of this via the existing `(*)` selector.

## The `@types/react` gap

`react@19.2.x` (the version pinned via `@types/react`) does not yet export `ViewTransition` — it exists only in Next's vendored `react-builtin`, which Next aliases `react` to inside `app/` at build/runtime. Runtime resolution is therefore correct, but `tsc --noEmit` (part of `pnpm types:check`) fails without a fix. `types/react-view-transition.d.ts` is a stopgap ambient module augmentation declaring `ViewTransition`'s prop shape on `'react'`. **Delete this file once `@types/react` ships real types** — a duplicate declaration will then produce a type error, which is the signal that the shim is no longer needed.

## Status: experimental, with a one-flag rollback

`experimental.viewTransition` is marked "not recommended for production" upstream. We're accepting this because it degrades gracefully: without browser support, or with the flag removed, navigation works exactly as before, just without the crossfade. The blast radius is small and easily revertible — `next.config.mjs`, `app/global.css`, the `<ViewTransition>` wrapper in `page.tsx`, and the sidebar's `style` prop in `layout.tsx`. To roll back, set `experimental.viewTransition` to `false` (or remove it); `<ViewTransition>` and `viewTransitionName` become inert no-ops with no other code changes required.

## Verification status

`pnpm types:check && pnpm lint` passed after every change. The dev server was restarted (required to pick up the `next.config.mjs` flag) and smoke-tested via HTTP requests to `/`, `/dictionary`, `/dictionary/baya`, and `/graph` — all returned 200 with no server-side errors related to `ViewTransition`. Home↔docs crossfade was confirmed working interactively in-browser after the `docs-content`-named home wrapper was adopted. Remaining **interactive** browser verification — sidebar non-flicker during docs-to-docs navigation, search-triggered navigation, `prefers-reduced-motion` emulation, and the canvas/graph/Base table frozen-frame check described below — has **not yet been performed** and remains an open follow-up. The vault currently generates no canvas or `base: true` pages, so the frozen-frame check in particular couldn't be exercised even with a browser available. This is not a blocker for merging given the rollback path above, but should not be forgotten.

## Deferred follow-ups

- **Directional (forward/back) slides for the prev/next footer** — Fumadocs' `Footer` component doesn't expose a `transitionTypes` hook on its internal links; would need a custom footer slot.
- **Shared-element morphs** — e.g. home page section cards or graph nodes morphing into their target pages. Requires coordinating matching `<ViewTransition name="...">` identifiers across independent components.
- **Theme-toggle circular reveal** — a different pattern (manual `document.startViewTransition()` on click), independent of `experimental.viewTransition`.
- **Canvas/graph/Base page exemption** — `(docs)/[...slug]` also serves canvas (React Flow), graph (React Force Graph), and Base table/gallery pages inside the same `docs-content` `<ViewTransition>`. Canvas/WebGL surfaces are captured as a static snapshot for the transition, which can look like a brief frozen frame. If the verification above finds this jarring, the fix follows ADR-0005's pattern: a frontmatter/page-data flag for the affected page kind, with `default="none"` for those pages specifically. Not pre-emptively implemented.
